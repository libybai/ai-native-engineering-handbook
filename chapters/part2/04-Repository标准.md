# 第 04 章 Repository 标准

> **Repository 是单个项目的唯一可信来源，也是 AI 进入项目后的基本理解单位。**

---

# 为什么

第 03 章已经定义：

Workspace 是整个研发环境的组织规范。

它负责组织多个 Repository、共享资源和本地运行环境。

但当 AI 真正进入一个项目时，它面对的不是整个 Workspace。

而是一个具体的 Repository。

因此，一个关键问题出现了：

> **如何设计一个能够让 AI 第一次进入项目就开始工作的 Repository？**

传统 Repository 通常以代码为中心。

它可能包含：

* 源代码
* 配置文件
* 构建脚本
* 测试代码
* 少量说明文档

这些内容足以让熟悉项目的人继续开发。

但对于第一次进入项目的人或 AI 来说，通常仍然不够。

AI 需要知道：

* 项目是什么？
* 项目为什么存在？
* 当前做到哪里？
* 架构如何组织？
* 近期任务是什么？
* 有哪些历史决策？
* 如何开始工作？

如果这些信息只存在于聊天记录、个人记忆或临时文档中，AI 每次进入项目都必须重新建立理解。

这会带来三个问题：

* 上下文重复解释。
* 项目知识无法沉淀。
* AI 输出难以长期保持一致。

因此，AI Native Repository 的目标不是只保存代码。

而是让 Repository 成为项目的长期知识中心。

---

# 核心思想

Repository 不只是代码仓库。

在 AI Native 工作流中，Repository 应该能够完整描述一个项目。

它是单个项目的：

**Single Source of Truth（唯一可信来源）。**

这意味着：

项目中长期有效的信息，都应该能在 Repository 中找到。

对于 AI 而言，一个合格的 Repository 应该具备自解释能力。

也就是：

**AI Self-Describing Repository（AI 可自解释仓库）。**

AI 可自解释仓库并不要求目录复杂。

也不要求把所有内容都写成大型文档。

它强调的是：

进入仓库后，人和 AI 都能够快速建立项目理解，并知道下一步应该查看什么、遵守什么、继续做什么。

Repository 与 Workspace、Runtime 的职责边界如下：

```text
Workspace
    负责组织整个研发环境

Repository
    负责组织单个项目

Runtime
    负责本地运行环境
```

本章只讨论单个 Repository。

不重新定义 Workspace。

也不讨论本地 Runtime。

---

# 标准规范

AI Native Repository 应同时包含：

* 代码
* 文档
* 规范
* 决策
* 上下文
* 历史记录

这些内容共同构成项目的长期知识。

## MUST

Repository 必须能够让新的 AI 第一次进入项目后，理解：

* 项目目标
* 当前状态
* 基本架构
* 工作规范
* 当前任务

如果一个 Repository 只能让 AI 看见代码，却不能理解项目目标、边界和当前状态，那么它还不是 AI Native Repository。

## SHOULD

Repository 应该长期保持自解释能力。

所有重要知识应沉淀到仓库。

不要只保存在：

* 聊天记录
* 个人笔记
* 临时会议纪要
* 某个开发者的记忆

这些信息可以作为过程材料。

但只要它们对项目长期有效，就应该进入 Repository。

## MAY

Repository 可以根据项目规模增加：

* `ADR`
* `docs/`
* `examples/`
* `scripts/`
* `tests/`
* `specifications/`

但不要为了形式而堆叠目录。

目录越多，不代表项目越规范。

真正重要的是：

每个目录和文件都有明确职责，并且会被持续维护。

## 长期文件职责

AI Native Repository 至少应明确以下长期文件的职责。

### README.md

`README.md` 是项目入口文件。

它用于让人和 AI 快速了解项目基本信息。

它应回答：

* 这个项目是什么？
* 解决什么问题？
* 当前如何开始阅读？
* 重要文档在哪里？

`README.md` 不应该承载所有细节。

它更像项目导航。

### PROJECT_CONTEXT.md

`PROJECT_CONTEXT.md` 是项目上下文文件。

它用于让 AI 理解项目目标、背景、边界、现状和约束。

它应回答：

* 项目为什么存在？
* 当前阶段目标是什么？
* 哪些事情在范围内？
* 哪些事情不在范围内？
* 当前有哪些关键约束？

对于 AI 协作来说，`PROJECT_CONTEXT.md` 是最重要的上下文入口之一。

### ARCHITECTURE.md

`ARCHITECTURE.md` 是架构说明文件。

它用于描述系统结构、模块关系、核心技术选择和架构边界。

它应回答：

* 系统由哪些主要模块组成？
* 模块之间如何协作？
* 核心技术选择是什么？
* 哪些架构边界不能随意突破？

它不需要替代代码。

它负责解释代码背后的结构。

### DECISIONS.md

`DECISIONS.md` 是决策记录文件。

它用于记录重大设计选择、原因和影响。

它应回答：

* 做过哪些重要决定？
* 为什么这样决定？
* 当时有哪些备选方案？
* 这个决定带来了什么影响？

没有决策记录，AI 很容易重复提出已经被否定的方案。

### TASKS.md

`TASKS.md` 是任务管理文件。

它用于记录当前任务、短期计划和待办事项。

它应回答：

* 当前正在做什么？
* 下一步准备做什么？
* 哪些任务已经完成？
* 哪些任务仍在等待？

它不是复杂项目管理系统的替代品。

它的价值在于让 AI 快速理解当前工作状态。

### CHANGELOG.md

`CHANGELOG.md` 是变更记录文件。

它用于记录项目演进历史。

它应回答：

* 项目发生过哪些重要变化？
* 哪些内容被新增、修改、移除或修复？
* 当前版本与之前版本有什么区别？

对于长期协作来说，变更历史是理解项目演进的重要线索。

---

# 实施方法

建立 AI Native Repository，可以从以下步骤开始。

**第一步：明确 Repository 的项目边界**

先回答：

* 这个 Repository 对应哪个项目？
* 它是否有独立生命周期？
* 它是否有独立版本？
* 它是否有独立发布或交付目标？

如果答案不清晰，说明项目边界还没有稳定。

不要急着增加目录。

先明确这个 Repository 存在的理由。

---

**第二步：建立项目入口**

确保 `README.md` 能够作为项目入口。

它不需要很长。

但必须能把读者带到正确位置。

一个好的入口文件应该告诉 AI：

* 先读什么。
* 再读什么。
* 哪些文件定义规则。
* 哪些文件记录当前状态。

---

**第三步：补齐长期上下文**

建立或维护 `PROJECT_CONTEXT.md`。

将项目目标、背景、边界、现状和约束写入其中。

不要让这些信息只存在于一次性对话中。

AI 每次进入项目时，都应该能从仓库中恢复基本上下文。

---

**第四步：沉淀架构和决策**

使用 `ARCHITECTURE.md` 说明系统结构。

使用 `DECISIONS.md` 记录重大决定。

这两个文件帮助 AI 理解：

* 为什么当前结构是这样。
* 哪些设计不能随意改变。
* 哪些方案曾经被讨论过。

---

**第五步：维护当前任务和变更历史**

使用 `TASKS.md` 记录当前任务和短期计划。

使用 `CHANGELOG.md` 记录项目演进。

这样，AI 不只知道项目是什么。

也知道项目正在往哪里走。

---

# 推荐结构

一个基础 AI Native Repository 可以从以下结构开始：

```text
repository/
├── README.md
├── PROJECT_CONTEXT.md
├── ARCHITECTURE.md
├── DECISIONS.md
├── TASKS.md
├── CHANGELOG.md
└── src/
```

如果项目规模更大，可以按需增加：

```text
repository/
├── README.md
├── PROJECT_CONTEXT.md
├── ARCHITECTURE.md
├── DECISIONS.md
├── TASKS.md
├── CHANGELOG.md
├── docs/
├── examples/
├── scripts/
├── specifications/
├── tests/
└── src/
```

这只是推荐结构。

不是固定模板。

真正的标准不是目录数量。

而是每个长期文件是否承担明确职责。

完整模板应放到第 22 章《模板中心》。

本章只说明这些文件为什么存在、承担什么职责。

# 示例

假设一个新 AI 第一次进入某个产品项目。

如果 Repository 只有代码，它可能需要先猜测：

* 项目目标是什么。
* 哪些模块是核心模块。
* 当前任务是什么。
* 哪些设计不能改。
* 最近发生过什么变化。

这会让 AI 在没有完整上下文的情况下开始工作。

结果通常是：

* 修改方向偏离目标。
* 重复提出旧方案。
* 忽略现有架构边界。
* 无法判断任务优先级。

如果 Repository 具备自解释能力，AI 的阅读路径可以是：

```text
README.md
    ↓
PROJECT_CONTEXT.md
    ↓
ARCHITECTURE.md
    ↓
DECISIONS.md
    ↓
TASKS.md
    ↓
CHANGELOG.md
    ↓
代码与测试
```

这样，AI 在写代码之前，已经理解了项目目标、约束、结构、决策和当前任务。

这就是 AI Self-Describing Repository 的价值。

---

# 最佳实践

建议遵循以下实践。

* 让 `README.md` 成为项目导航，而不是杂乱说明集合。
* 将项目目标和边界写入 `PROJECT_CONTEXT.md`。
* 将结构性设计写入 `ARCHITECTURE.md`。
* 将重大选择写入 `DECISIONS.md`。
* 将当前任务写入 `TASKS.md`。
* 将长期变化写入 `CHANGELOG.md`。
* 让长期有效的信息进入 Repository。
* 定期清理过期文档。

维护 Repository 时，要避免两个极端。

第一个极端是：

只有代码，没有上下文。

第二个极端是：

目录很多，但没人维护。

AI Native Repository 追求的是：

少量稳定文件。

明确职责。

持续更新。

# 常见错误

**错误一：把 Repository 标准写成 Git 教程。**

Repository 标准解决的是项目如何自解释。

不是 Git 命令如何使用。

分支、提交、评审和发布相关内容，应放到第 06 章《Git 工作流》。

---

**错误二：只有代码，没有项目上下文。**

AI 可以阅读代码。

但代码通常不能完整回答项目目标、业务边界和当前阶段。

缺少上下文时，AI 很容易做出局部正确但整体错误的修改。

---

**错误三：把重要决策留在聊天记录里。**

聊天记录适合过程沟通。

但不适合作为长期可信来源。

重大决定应该进入 `DECISIONS.md`。

---

**错误四：把当前任务只放在个人待办里。**

个人待办无法帮助新的 AI 理解项目状态。

当前任务和短期计划应进入 `TASKS.md`。

---

**错误五：为了规范而堆叠目录。**

目录越多，维护成本越高。

如果某个目录没有明确职责，也没有持续维护计划，就不应该加入。

---

# 本章总结

Repository 是 Workspace 的组成单位。

Workspace 负责组织整个研发环境。

Repository 负责组织单个项目。

Runtime 负责本地运行环境。

在 AI Native 工作流中，Repository 不只是代码仓库。

它是单个项目的唯一可信来源。

一个合格的 AI Native Repository，应该是：

**AI Self-Describing Repository。**

也就是 AI 可自解释仓库。

它应通过少量稳定的长期文件，帮助人和 AI 快速理解：

* 项目目标
* 当前状态
* 基本架构
* 历史决策
* 当前任务
* 项目变化

只有 Repository 具备自解释能力，AI 才能从一次性辅助工具，真正转变为能够长期参与项目的软件研发协作者。
